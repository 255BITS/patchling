# smartapply deterministic-apply benchmark

`smartapply` now applies every file's patch **deterministically first** (no LLM
call) via `applyPatchToFile`, and only falls back to the LLM for files whose
patch does not apply cleanly. Each file is processed in parallel, so a diff
touching ten files where nine apply cleanly makes a single LLM call instead of
ten. Pass `{ forceLlm: true }` to disable the fast path.

This directory measures how often the deterministic path wins.

## Files

- `diff-corpus.json` — 128 generated `(original, diff, expected)` examples
  across 13 categories (single-line, multi-hunk, creations, deletions,
  multi-file, whitespace, append/prepend, no-trailing-newline, headerless
  LLM-style, fuzzy context mismatch, malformed markers, large refactors, and
  css/yaml/html edge cases). Generated by a fan-out of 16 agents.
- `run.js` — applies each example with the same per-file logic as `smartapply`'s
  fast path and classifies every file-patch as:
  - **clean-correct** — applied with no LLM and matched the expected output
  - **clean-wrong** — applied with no LLM but produced wrong output (silent
    corruption — the dangerous case)
  - **fallback** — could not apply (returned `null`), so `smartapply` calls the LLM

Run it with `npm run benchmark` (or `node benchmark/run.js [corpus.json]`).

## Current results (128 examples, 135 file-patches)

| outcome        | files | share |
| -------------- | ----- | ----- |
| clean-correct  | 110   | 81.5% |
| clean-wrong    | 5     | 3.7%  |
| fallback → LLM | 20    | 14.8% |

**~81% of files apply with zero LLM calls.** The remaining ~15% safely fall back.

The 5 "clean-wrong" rows are not silent corruption of valid diffs:

- 3 are the `malformed-markers` category — diffs with literal `++`/`--` body
  markers (e.g. `++  console.error(msg)`). A `+` line can legitimately add text
  that itself starts with `+`, so there is no deterministic way to know the diff
  is malformed; these need the LLM but the applier can't tell. (Inherent.)
- 2 (`multi-file-ts-context-mismatch-fallback`, `no-trailing-newline-delete-trailing-line`)
  are **generator errors** — the LLM that produced the fixture computed an
  `expected` that does not match its own diff. The applier output is correct.

On valid diffs, silent corruption is effectively **zero**.

## Bugs this benchmark surfaced (now fixed)

1. **Headerless parser misread body lines as file headers.** In
   `parseDiffPerFile`'s headerless path, the `---`/`+++` detection (`{2,3}`
   markers) matched a hunk body line that deletes/adds content starting with
   `-`/`+` (e.g. a markdown list item `- [ ] x` appears as `-- [ ] x`), silently
   truncating the patch. Fixed by only treating a `--`/`---` line as a from-header
   when the next line is a `++`/`+++` to-header (unified diffs always pair them).

2. **The applier trusted `@@` line numbers.** It jumped to `origStart - 1` and
   failed if the context did not match there. LLM-generated diffs routinely get
   line numbers slightly wrong, which caused most multi-hunk and
   no-trailing-newline fallbacks. `applyPatchToFile` now locates each hunk by
   matching its full context+deletion block (line numbers are only a tie-break
   hint), like `patch`/`git apply` fuzz. The full block must still match exactly,
   so unapplicable hunks return `null` rather than corrupting the file. This
   raised clean-correct from ~68% to ~81%.

3. **Blank-line additions were swallowed after a newline-less last line.** Fixed
   by applying on newline-free lines joined with `\n`.

## Caveat

The corpus is LLM-generated, so a few examples have inconsistent `expected`
fields (see above). The numbers are a guide, not ground truth. Regenerate or
hand-audit before treating any single percentage as exact.
